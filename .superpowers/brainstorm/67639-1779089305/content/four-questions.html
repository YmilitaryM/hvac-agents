四大关键问题的设计方案

时延处理 · 冷负荷预测 · Agent分工与辩论 · 编排管理

Q1: 时延问题 — 分层异步 + 消息总线

层级 典型时延 处理方式
实时控制层< 100ms同步执行，不等待上层
策略执行层秒级定时任务触发，前置条件实时检查
RL审核引擎< 100ms本地模型推理，无需网络调用
LLM策略生成1~30秒提前生成，异步执行，不阻塞实时控制
人工审核分钟~小时非阻塞，策略提前生成，审核不影响实时运行
RL模型训练小时~天离线训练，定期更新模型权重
核心设计原则：
  • 消息总线解耦 — 各层通过 Redis/MQTT 异步通信，互不阻塞
  • 实时层完全自治 — PID、安全保护不依赖上层，自己闭环
  • 策略预生成 — LLM提前N小时生成策略（利用负荷预测），不是临到时间才生成
  • RL本地推理 — 模型部署在本地，推理无网络延迟
  • 超时降级 — LLM调用超时？用上次的策略模板 + 当前数据填充

Q2: 冷负荷预测 — 混合模型 + 多时间尺度

时间尺度 方法 用途
超短期 (5~15min) 时序模型 (LightGBM/XGBoost) 实时控制参数微调
短期 (1~6小时) 深度学习 (TFT/LSTM + 注意力) 策略生成、加减机决策
中长期 (24h~72h) 物理模型 + 天气预报 + 日历特征 日前调度计划、蓄冷策略
特征输入： 室外干球温度 · 相对湿度 · 太阳辐照度 · 时间特征(小时/星期/月份) · 建筑 occupancy 模式 · 历史负荷序列 · 电价时段 仿真阶段： 用建筑热工模型 (RC网络 或 EnergyPlus 简化模型) 生成合成负荷数据 → 训练ML模型 → 后期接入真实BA数据后 finetune

Q3: Agent 分工与辩论机制

6个Agent + 辩论仲裁机制
Agent职责输入输出
🔍 监测Agent设备实时状态、异常检测、健康评估传感器数据流异常告警、设备健康分
📈 预测Agent多时间尺度冷负荷预测天气+历史+日历负荷预测值+置信区间
🎯 策略Agent生成运行策略(加减机、模式切换)预测+设备状态+电价策略方案+预期效果
⚙️ 参数Agent设备运行参数优化(水温设定、频率)策略+实时负荷+设备模型最优参数组合
🛡️ 安全Agent硬约束校验、安全边界守护任何Agent的输出通过/拦截+原因
👑 协调Agent编排流程、仲裁冲突、汇总结果所有Agent输出最终决策+置信度
辩论机制的三种模式：
模式A: 并行提案+投票

策略Agent和参数Agent各提案，协调Agent加权投票，适用于常规工况

模式B: 对抗辩论

一个Agent提案，另一个扮演"反方"质疑，适用于复杂/边界工况

模式C: 串行接力

预测→策略→参数→安全，链式传递，适用于负荷变化平缓的时段

Q4: Agent 编排管理 — LangGraph StateGraph + 黑板模式

编排机制：
  • LangGraph StateGraph — 定义Agent之间的调用拓扑和条件分支
  • 共享黑板 (Shared State) — 所有Agent读写同一个结构化状态对象，避免点对点通信爆炸
  • 协调Agent = Supervisor — 决定"下一步调用哪个Agent"，类似 LangGraph 的 Supervisor 模式
  • 安全Agent = 全局拦截器 — 任何Agent的输出先经过安全Agent再进入下一环节
Agent间通信方式：
  • 不直接对话 — Agent之间不互相调用，只通过共享状态传递信息
  • 事件总线 — 异常事件（如设备故障）通过 Pub/Sub 广播，订阅者自主响应
  • 结构化输出 — 每个Agent的输出是预定义的 Schema（Pydantic模型），保证下游可解析
运行时管理：
  • 每个Agent独立进程/协程，可独立扩缩、重启、升级
  • 心跳检测 + 超时降级
  • 全链路追踪（每次决策的完整Agent调用链可回溯）

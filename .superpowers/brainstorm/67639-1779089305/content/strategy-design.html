算法枚举可行解 + LLM做上下文推理，各司其职
Step 1: 触发分类
是什么触发了策略生成？
确定策略类型和审核级别
→
Step 2: 约束提取
从监测Agent+安全Agent
获取当前设备约束边界

Step 3: 方案枚举
优化求解器枚举可行设备组合
每个方案计算COP/成本/碳排

Step 4: LLM选择
Deep LLM基于上下文推理

Step 5: 结构化输出
Pydantic Schema约束
生成完整策略对象
为什么Step 3不能用LLM做？
制冷站设备组合是组合优化问题。中型站(5台冷机+5台冷却塔+10台泵)的组合空间巨大，且每台设备有自己非线性的COP曲线和约束（最小流量、喘振边界、水泵最小频率）。LLM不擅长穷举和精确数值计算——这必须由优化求解器完成。LLM的价值在Step 4：从求解器给出的Top-3方案中，根据天气预报置信度、设备健康趋势、历史策略效果等"软信息"做最终选择。

策略生命周期

状态谁操作做什么下一状态
draft 策略Agent生成策略草案→ under_review
under_review 3 Advocate + 协调Agent辩论审查→ approved / rejected
approved RL审核引擎置信度评估 → 自动批或推人工→ scheduled
rejected —记录驳回原因，写入Memory供RL学习终端状态
scheduled 调度引擎注册定时任务，等待触发时间→ executing / expired
executing 调度引擎 + 安全Agent前置条件检查 → 逐条下发指令 → 监控执行→ completed / aborted
completed Memory Log记录实际效果，触发RL+反思终端状态
aborted Memory Log记录失败原因(前置条件不满足/执行异常)，触发反思终端状态

策略的Memory机制 — 借鉴TradingAgents的延迟反思

每次策略执行后，两路反馈注入Memory：

RL信号（数值）

实际COP - 预期COP = 奖励
实际节能量 - 预期节能量 = 奖励
→ 更新RL模型权重

LLM反思（文本）

"2026-05-15减机策略：COP从4.8升至5.3(+10.4%)，低于预期的+12%。原因是冷却塔逼近度在单机高负荷下上升了0.8°C，未在策略中考虑。下次类似工况需增加冷却塔风扇调速的联动动作。"

LLM反思文本被注入到策略Agent的未来Prompt中，类似TradingAgents的 past_context 机制。当策略Agent下次遇到相似工况，Prompt中会包含："历史相似工况下，减机后COP平均提升10.3%，但需注意冷却塔逼近度上升问题——建议联动调节冷却塔风扇。"

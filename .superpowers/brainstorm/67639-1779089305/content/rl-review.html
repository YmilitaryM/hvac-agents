强化学习驱动的策略审核进化系统

从"人工全审"到"RL辅助审核"再到"RL自动审核"，逐步减轻人工负担

RL 审核进化三阶段

Phase 1

人工全审
每一条策略人工审核
RL收集数据学习偏好
约1-2个月数据积累
→

Phase 2

RL辅助审核
高置信度策略自动过
低置信度推人工确认
持续学习优化中
→

Phase 3

RL自动审核
常规策略自动执行
异常/高风险推人工
人工只看边缘case

RL 审核引擎 — 核心设计

设计要素 方案 说明
RL算法 Contextual Bandit / Offline RL 安全性优先；不随机探索，从历史审核数据中学习
状态空间 冷负荷、室外温湿度、电价时段、设备运行状态、策略类型 描述"当前工况"的完整特征向量
动作空间 {自动批准, 推送人工审核, 建议修改} 离散决策，不是连续控制
奖励信号 节能效果 + 舒适度维持 + 设备健康 + 人工审批匹配度 多目标加权，从执行后反馈中学习
安全护栏 硬约束规则引擎始终在线 RL不能让"关掉所有机组"这种策略通过，规则引擎是第一道关

学习闭环

1. LLM生成策略 → 2. RL审核引擎评估 (当前工况下该策略置信度?) → 置信度>阈值 → 自动批准执行 → 置信度<阈值 → 推送人工 3. 执行结果反馈 → 能耗降低了多少? 舒适度是否保持? → 4. 奖励信号回传RL模型 → 更新模型参数 → 5. 下次相似工况，RL审核更准 6. 人工审核决策也作为训练信号 → 操作员通过/驳回 → 反馈RL模型 → 减少未来同类策略的审核负担

安全设计：RL的边界

硬护栏（不可逾越）

  • 规则引擎始终在RL之前执行 — 违反安全约束的策略直接拒绝，不进入RL审核环节
  • RL永远不能修改策略内容本身，只能决定"通过/推送人工/建议修改"
  • 极端工况（超高温、设备故障、安全事件）强制推送人工，RL置信度再高也无效
  • 每个自动执行的策略都有"紧急停止"按钮，操作员可随时介入

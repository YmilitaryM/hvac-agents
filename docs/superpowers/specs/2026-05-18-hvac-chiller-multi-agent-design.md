## 1. 系统目标

构建 HVAC 制冷站多智能体系统，实现三大核心功能：

- **实时监测** — 系统设备运行状况、异常检测、设备健康评估
- **冷负荷预测** — 多时间尺度实时预测系统所需冷负荷
- **智能控制** — 运行策略生成、设备参数优化、策略审核与执行

运行模式：先在仿真环境下开发和验证，后期可切换到真实物理系统对接。

## 2. 总体架构

五层架构，数据自下而上流动，决策自上而下传递：

```
L5 智能层    — Agent 决策引擎 (LangGraph 编排，LLM 推理，辩论仲裁)
L4 服务平台层 — 报告服务、RAG 标准库、RL 审核引擎、定时调度器、消息总线
L3 实时控制层 — PID 调节、设备启停、安全联锁、故障切换、数据采集网关
L2 仿真层    — 冷机模型、冷却塔模型、水泵模型、管路水力模型、建筑热工模型
L1 基础设施层 — 时序DB、向量DB、消息队列、对象存储、监控
```

### 核心设计原则

- **LLM 不做实时控制** — 确定性算法负责毫秒级控制，LLM 负责策略推理和智能分析
- **人在回路中** — 关键策略需人工审核，RL 逐步减轻审核负担
- **硬约束不可逾越** — 安全 Agent 是规则引擎（非 LLM），硬约束在任何决策之前执行
- **每层自治** — 上层故障不影响下层运行，实时控制层完全独立
- **全链路可追溯** — 每条策略记录生成、审查、执行、反馈的完整生命周期

## 3. Agent 体系

共 **10 Agent + 1 优化引擎**，按功能（非学科）分三层：

### 感知层 — 数据→信息

| Agent | LLM 等级 | 职责 | 工具 |
|-------|---------|------|------|
| 监测 Agent | Quick | 设备状态、异常检测、健康评估、性能曲线退化追踪 | 传感器接口、时序查询 |
| 预测 Agent | Quick | 多尺度冷负荷预测（超短期/短期/中长期） | 天气 API、ML 推理、历史数据 |
| 策略 Agent | Deep | 调用联合优化引擎获取帕累托解集 → LLM 推理选择 → 生成完整策略（含设备组合、参数、过渡计划） | 联合优化引擎、设备模型、电价数据、Memory Log |

### 辩论层 — 三张力多视角验证

按制冷站运行中 **天然对立的优化目标** 划分，每个 Advocate 通过 RAG 获取所需的多学科知识：

| Agent | LLM 等级 | 代表立场 | 审查维度 |
|-------|---------|---------|---------|
| 可靠性 Advocate | Deep | 设备安全 + 供冷可靠性 | 安全余量、设备健康、喘振风险、启停频率、热力学可行性 |
| 效率 Advocate | Quick | 运行成本 + 能耗最优 | COP 优化、电价响应、需量管理、设备负荷分配 |
| 合规 Advocate | Quick | 碳排放 + 法规合规 | 碳排放核算、碳配额管理、标准合规性 |

### 决策层 — 仲裁→执行

| Agent | 类型 | 职责 |
|-------|------|------|
| 协调 Agent | Deep LLM | 辩论仲裁、加权决策、冲突解决 |
| 安全 Agent | 规则引擎 | 硬约束校验（非 LLM）、全局拦截 |
| 参数 Agent | 控制器 | 策略执行期间实时微调（±5% 范围内）、死区控制、速率限制 |
| 报告 Agent | Quick LLM | 多类型报告生成（只读消费者，不参与决策闭环） |

### 联合优化引擎（新增，非 Agent）

| 组件 | 类型 | 职责 |
|------|------|------|
| 联合优化引擎 | 数值计算服务 | MINLP 求解 + 性能曲线管理 + 帕累托前沿生成 |

## 4. 工作流与辩论机制

### 7 阶段工作流

```
[触发] 定时 / 负荷变化超阈值 / 故障 / 电价信号 / 手动

Phase 1 — 感知层（并行）
  监测 Agent → 设备状态快照 + 约束边界
  预测 Agent → 多尺度负荷预测 + 置信区间

Phase 2 — 策略生成
  策略 Agent ← 消费 Phase 1 输出 + Memory Log
  → 调用联合优化引擎 → 获取帕累托解集
  → Deep LLM 推理选择 → 输出完整策略方案

Phase 3 — 辩论审查（并行）
  可靠性 Advocate + 效率 Advocate + 合规 Advocate
  → 各自从立场审查策略 → 输出结构化审查意见

Phase 4 — 仲裁
  协调 Agent ← 汇总 3 方意见
  → 无冲突：直接通过
  → 有冲突：触发定向辩论（max 2 轮）→ 加权仲裁

Phase 5 — 安全校验
  安全 Agent → 硬约束检查（喘振边界、最小运行时间、电气约束...）
  → 通过 / 拦截

Phase 6 — 审核与执行
  RL 审核引擎 → 置信度评估
  → 高置信度：自动批准
  → 低置信度：推送人工审核
  → 调度引擎注册定时任务 / 即时下发

Phase 7 — 反馈闭环
  执行结果 → RL 奖励信号（更新模型权重）
  → LLM 反思（写入 Memory Log，注入未来 Prompt）
```

### 辩论机制的三种模式

| 模式 | 适用场景 | 方式 |
|------|---------|------|
| 并行审查 | 常规工况 | 3 Advocate 同时审查，协调 Agent 汇总 |
| 定向辩论 | 存在冲突 | 冲突双方 2 轮辩论，协调 Agent 仲裁 |
| 加速通道 | 故障响应 | 跳过辩论，安全 Agent + 实时控制层直接动作 |

### 不同工况的审核强度

| 策略类型 | 审核强度 | 时延要求 |
|---------|---------|---------|
| 设备轮换 | 轻量审查 | 分钟级 |
| 负荷跟随 | 常规审查 | 分钟级 |
| 需求响应 | 常规审查 | 分钟级 |
| 模式切换 | 严格审查（全票通过） | 10 分钟级 |
| 故障响应 | 加速通道 | 毫秒级 |

## 5. 策略生成与联合优化

### 策略的完整结构

每条策略包含：

1. **终态目标** — 设备组合 + 目标参数
2. **过渡计划** — 分阶段斜坡 + 稳定性检查点 + 启动联锁序列
3. **前置条件** — 执行前必须满足的条件
4. **中止条件** — 什么情况下终止执行
5. **回滚计划** — 终止后如何恢复
6. **预期效果** — COP 提升、节能量、碳减排
7. **LLM 推理** — 为什么从帕累托解中选了这个

### 联合优化引擎

**问题本质：MINLP（混合整数非线性规划）**

- 整数变量：哪台设备开/关
- 连续变量：每台设备的设定值、频率、阀门开度
- 非线性约束：冷机 COP 曲面、泵相似定律、冷却塔热力特性、喘振边界

**约束条件**：

| 约束类型 | 内容 |
|---------|------|
| 喘振边界 | PLR_i ∈ [PLR_min,surge(T_cw), 1.0] ∪ {0}，喘振区随冷凝温度升高而扩大 |
| 最小运行/停机时间 | 启动后 ≥ MinRuntime，停机后 ≥ MinOffTime |
| 电机启动间隔 | 大电机启动间隔 ≥ 30s，防止电气冲击 |
| 流量约束 | 蒸发器/冷凝器最小流量 |
| 温度约束 | 冷冻水供水温度范围、冷却水温度范围 |
| 设备可用性 | 故障/维护中的设备不可选 |

**目标函数**：

```
min(电费 + 碳成本 + 启停磨损成本)
其中：启停磨损成本 = Σ(每台设备本次启停次数 × 单次磨损成本)
```

**优化流程**：

```
Step 1: 负荷特征化 → 需求容量范围
Step 2: 系统建模 → 加载设备代理模型（多项式性能曲线）
Step 3: MINLP 联合求解 → 设备组合 + 参数一体优化
Step 4: 帕累托前沿 → 成本-碳排-可靠性三维非劣解集（3~5个方案）
Step 5: LLM 选择 → 策略 Agent 从帕累托解中推理选择
Step 6: 结构化输出 → 完整策略包（含过渡计划）
```

### 策略生命周期

```
draft → under_review → approved/rejected → scheduled → executing → completed/aborted
```

每个状态转换可追溯，是 RL 学习和审计的基础。

### 策略 Memory 机制（借鉴 TradingAgents 延迟反思）

每次策略执行后，两路反馈：

- **RL 信号（数值）** — 实际 COP - 预期 COP = 奖励 → 更新模型权重
- **LLM 反思（文本）** — 策略效果分析和教训 → 写入 Memory Log → 注入策略 Agent 未来 Prompt

## 6. 性能曲线获取

**不依赖厂家样本曲线**，采用三层自适应获取：

| 层 | 方法 | 用途 | 更新 |
|---|------|------|------|
| 物理机理模型 | 逆卡诺循环 + Merkel 理论等第一性原理 | 仿真阶段生成合成数据 | 一次构建 |
| 代理模型 (Surrogate) | 多项式回归拟合 | 优化引擎实时使用（微秒级计算） | 离线拟合 |
| 在线参数辨识 | 递推最小二乘(RLS) / 卡尔曼滤波 | 接入实机后持续修正系数，追踪老化 | 持续在线 |

冷机 COP 曲面的代理模型形式：

```
COP(PLR, T_chw, T_cw) = a₀ + a₁·PLR + a₂·PLR² + a₃·T_chw + a₄·T_cw + a₅·T_cw² + a₆·PLR·T_cw
```

系数 [a₀..a₆] 由仿真数据回归或运行数据在线辨识得到。

## 7. 稳定性与过渡控制

### 三道防线

| 防线 | 位置 | 机制 | 响应时间 |
|------|------|------|---------|
| 策略级 | 策略 Agent + 优化引擎 | 过渡计划内置斜坡速率、分阶段执行、稳定性检查点、预置回滚方案 | 策略生成时 |
| 执行级 | 参数 Agent | 死区(±0.3°C)、速率限制(0.1°C/min)、抗积分饱和、振荡检测 | 秒级 |
| 保护级 | 安全 Agent | 硬限值保护、过渡中止条件、最小运行/停机时间保护、防喘振保护 | 毫秒级 |

### 过渡计划结构

策略不再是"15:00 停 2 号机"的原子指令，而是多阶段过渡：

Phase 1 — 降负荷（斜坡速率限制）
Phase 2 — 稳定性观察（偏差检查）
Phase 3 — 继续降负荷至停机
Phase 4 — 最终稳定确认

每个阶段包含：持续时间、动作序列、稳定性检查条件。过渡失败时执行预置的回滚计划。

### 设备启动联锁序列

冷机启动不是单个指令，而是标准序列模板：

1. 开阀门（冷冻水+冷却水）
2. 启动水泵（冷冻+冷却），确认流量
3. 启动冷却塔风扇（低速）
4. 联锁检查（水流、油温、导叶位置）
5. 启动压缩机，缓慢加载至目标负荷率

电气约束：大电机启动间隔 ≥ 30s，错开冲击电流。

## 8. RL 审核引擎

### 三阶段进化

```
Phase 1 (1~2月): 人工全审 → RL 旁观学习
Phase 2 (持续):   RL 辅助 → 高置信度自动过，低置信度推人工
Phase 3 (成熟后): RL 自动审常规策略，人工只看高风险/异常 case
```

### 核心设计

| 要素 | 方案 | 说明 |
|------|------|------|
| 算法 | Contextual Bandit / Offline RL | 不随机探索，从历史审核数据中学习 |
| 状态空间 | 冷负荷、室外温湿度、电价时段、设备状态、策略类型 | 描述当前工况的特征向量 |
| 动作空间 | {自动批准, 推送人工审核, 建议修改} | 离散决策 |
| 奖励信号 | 节能效果 + 舒适度维持 + 设备健康 + 人工审批匹配度 | 多目标加权 |

### 安全护栏

- 规则引擎始终在 RL 之前执行 — 违反硬约束的策略直接拒绝
- RL 不能修改策略内容，只能决定"通过/推人工/建议修改"
- 极端工况强制人工，RL 置信度再高也无效
- 自动执行的策略有紧急停止机制

### 学习信号来源

1. **执行结果反馈** — 实际节能效果、COP 偏差、舒适度维持
2. **人工审核决策** — 操作员通过/驳回的偏好模式

## 9. 报告子系统

### 报告类型

| 报告 | 触发 | 核心内容 | 引用标准 |
|------|------|---------|---------|
| 运行日报 | 每日定时 | 设备运行时长、启停次数、能耗、异常事件 | GB/T 17981 |
| 能效周报/月报 | 每周/月定时 | 系统 COP/EER、各设备能效对比、节能效果 | GB 19577、GB 50189 |
| 碳排放月报 | 每月 + 合规节点 | 排放总量/强度、配额进度、减排效果 | ISO 14064、GB/T 32150 |
| 设备健康报告 | 每周 + 异常触发 | 性能衰减趋势、维护建议、故障风险排序 | GB/T 15316 |
| 策略审计报告 | 每策略 + 汇总 | 策略内容、审批链、预期 vs 实际效果、RL 置信度 | — |

### 生成管线

```
Phase 1 — 数据聚合：时序 DB + 事件 Log + 策略 Log + Memory
Phase 2 — KPI 计算：按标准公式计算 COP/EER/碳排放强度（纯函数，需单元测试）
Phase 3 — AI 分析：LLM 趋势分析、异常模式识别、对标评估
Phase 4 — 模板填充：结构化数据填入 + LLM 生成叙述 + 标准条款引用
Phase 5 — 多格式输出：Dashboard JSON / PDF / Excel / Markdown
```

### 标准引用机制

两次查询 RAG 标准库：
- Phase 2：查标准的具体数值用于 KPI 对标（如"GB 19577 规定的 COP 限定值"）
- Phase 4：查标准的条款编号和格式要求，嵌入报告正文

### 报告 Agent 定位

只读消费者，消费所有 Agent 的结构化输出但不回写，不参与决策闭环。

### 内部组件

| 组件 | 职责 | 技术 |
|------|------|------|
| 模板库 | 每种报告的结构化模板 | Jinja2 / Markdown |
| KPI 计算器 | 按标准公式计算指标 | Python 函数库（需单元测试） |
| 数据分析引擎 | 趋势识别、对标分析 | Quick LLM + 统计方法 |
| 标准引用器 | 查询 RAG 标准库 | 向量检索 + LLM 精排 |
| 叙述生成器 | 数据→自然语言 | Quick LLM + 结构化 Prompt |
| 格式渲染器 | 多格式输出 | WeasyPrint / Pandoc / openpyxl |
| 调度器接口 | 定时任务 + 按需查询 | 与 L4 调度引擎对接 |

## 11. 数据与基础设施

### 数据存储

| 数据类型 | 存储 | 用途 |
|---------|------|------|
| 传感器时序数据 | InfluxDB / TimescaleDB | 监测、预测、分析 |
| 设备性能曲线 | 代理模型系数表 | 优化引擎 |
| 策略/审核/Memory Log | PostgreSQL (结构化 JSON) | 审计、RL 学习、反思 |
| 事件/告警 | PostgreSQL | 报告、分析 |
| 标准文档 Embedding | 向量数据库 (Milvus/pgvector) | RAG 标准查询 |
| 报告文件 | 对象存储 (MinIO/S3) | 报告分发 |

### 消息总线

- 层间异步通信：Redis Pub/Sub 或 MQTT
- 实时控制层：独立于消息总线，自己的确定性控制循环
- LLM 调用超时降级：超时 → 用上次策略模板 + 当前数据填充

### 时延处理

| 层级 | 典型时延 | 处理方式 |
|------|---------|---------|
| 实时控制层 | < 100ms | 同步执行，不等待上层 |
| 策略执行层 | 秒级 | 定时任务触发，前置条件实时检查 |
| RL 审核引擎 | < 100ms | 本地模型推理 |
| LLM 策略生成 | 1~30s | 提前生成，异步执行 |
| 人工审核 | 分钟~小时 | 非阻塞，策略提前生成 |
| RL 模型训练 | 小时~天 | 离线训练，定期更新权重 |

## 12. 仿真层设计

### 设备物理机理模型

| 设备 | 建模方法 | 关键参数 |
|------|---------|---------|
| 离心冷机 | 逆卡诺循环 + 效率修正 | 額定制冷量、COP、喘振边界、部分负荷特性 |
| 冷却塔 | Merkel 理论 + NTU 法 | 传热单元数、风机功率曲线 |
| 水泵 | 相似定律 + 管路特性曲线 | 额定流量、扬程、效率曲线 |
| 阀门/管路 | 水力阻力模型 | Kv 值、管路阻力系数 |
| 建筑热工 | RC 网络模型 (3R2C) | 热容、热阻、太阳得热系数 |

### 数据生成流程

1. 物理模型跑大量仿真工况（覆盖负荷率 10%~100%，各种温度组合）
2. 生成合成数据集
3. 多项式回归 → 代理模型（轻量，供优化引擎使用）
4. 代理模型 + 模拟传感器噪声 → 仿真数据流
